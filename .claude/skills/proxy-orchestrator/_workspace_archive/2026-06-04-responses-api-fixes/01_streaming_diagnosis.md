## Bug 1: CRITICAL — SSE line splitting causes data loss on TCP fragmentation
**File:** `src/translate/stream/openai-to-anthropic.ts`
**Lines:** 229 (main loop), 210 (end-of-stream)
**Severity:** CRITICAL

### Problem
The parser splits incoming data on single `\n` at line 229:

```typescript
const lines = buffer.split('\n');
```

If a TCP packet boundary splits a `data:` line mid-content, the two fragments are parsed independently and both fail or are silently lost.

**Example failure scenario:**
```
Packet 1: data: {"choices":[{"index":0,"delta":{"content":"Hel
Packet 2: lo"}}]}\n\n
```

After packet 1:
- `buffer.split('\n')` → `["data: ...Hel"]` (no `\n` found)
- The incomplete JSON `data: ...Hel` is parsed → `JSON.parse` throws → silently skipped (`catch { continue; }`)

After packet 2:
- `buffer = "lo\"}}]}\n\n"`
- `buffer.split('\n')` → `["lo\"}}]}", ""]`
- Line `"lo\"}}]}"` does NOT start with `data: ` → silently skipped

**Result: The chunk `"Hello"` is entirely lost.**

### Compare with `anthropic-to-openai.ts` (correct implementation)
Line 149 in that file uses `buffer.split("\n\n")`, which correctly preserves partial lines across TCP chunks:

```typescript
const parts = buffer.split("\n\n");
buffer = parts.pop() || ""; // keep incomplete last part
```

### Fix
Replace `buffer.split('\n')` with `buffer.split('\n\n')` and re-structure the parsing loop to handle multi-line SSE frames. The fix should:

```typescript
// After buffer += chunk:
const frames = buffer.split('\n\n');
buffer = frames.pop() || ''; // keep incomplete frame

for (const frame of frames) {
  if (!frame.trim()) continue;
  const lines = frame.split('\n');
  for (const line of lines) {
    if (line.trim() && line.startsWith('data: ')) {
      // ... existing parsing logic
    }
  }
}
```

## Bug 2: CRITICAL — Missing `message_start` when stream has empty/null delta with finish_reason
**File:** `src/translate/stream/openai-to-anthropic.ts`
**Lines:** 26-40 (processStreamDelta entry), 70-85/125-139/172-186 (message_start emission)
**Severity:** CRITICAL

### Problem
If the OpenAI stream's first meaningful chunk is:

```
data: {"choices":[{"index":0,"delta":{},"finish_reason":"stop"}],"usage":{"prompt_tokens":5,"completion_tokens":3}}\n\n
```

The flow is:
1. `processStreamDelta({}, parsed)` is called
2. `delta` is `{}` → `delta.tool_calls?.length` is falsey, `delta.reasoning_content` is undefined, `delta.content` is undefined
3. No content block is started → `message_start` is never emitted
4. After stream end, `message_delta` and `message_stop` are emitted without a preceding `message_start`

The resulting Anthropic SSE stream would be:
```
event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"end_turn"},"usage":{...}}

event: message_stop
data: {"type":"message_stop"}
```

This violates the Anthropic streaming protocol, which requires `message_start` as the first event.

### Trigger conditions
- Any stream where the ONLY content-bearing chunk has an empty/null delta with `finish_reason`
- Examples: empty response with stop reason, content-filtered response, or response with only empty content

### Fix
At stream end, if `messageStarted` is false, emit a synthetic `message_start` before `message_delta`:

```typescript
// After the while loop, before close
if (isToolUse || hasStartedTextBlock || hasStartedThinkingBlock) {
  enqueueSSE(controller, "content_block_stop", { ... });
}

if (!messageStarted) {
  enqueueSSE(controller, "message_start", {
    type: "message_start",
    message: {
      id: messageId,
      type: "message",
      role: "assistant",
      content: [],
      model,
      stop_reason: null,
      stop_sequence: null,
      usage: { input_tokens: 0, output_tokens: 0 },
    },
  });
}

// Then message_delta and message_stop as before
```

## Bug 3: HIGH — Tool call arguments dropped when sent in same chunk as tool call ID
**File:** `src/translate/stream/chat-completions-to-responses.ts`
**Lines:** 240-261
**Severity:** HIGH

### Problem
When a tool call's `id`, `name`, and initial `arguments` arrive in the same streaming chunk:

```json
{"choices":[{"index":0,"delta":{"tool_calls":[{"index":0,"id":"call_abc","function":{"name":"get_weather","arguments":"{\"location\":\"SF\""}}]}}]}
```

The code takes the `if (tc.id)` branch (line 240) and calls `startToolCallItem()`. The `else if (tc.function?.arguments)` branch (line 244) is skipped because it's part of the same `if/else if` chain.

**The initial arguments `"{\"location\":\"SF\""` are silently lost.** Only subsequent argument-only chunks (without `id`) will be processed.

### Real-world impact
Several OpenAI-compatible providers send initial tool call arguments in the same chunk as the declaration. Those arguments are lost, causing malformed function call arguments in the Responses API output.

### Fix
Restructure the tool call handler to process both id/name AND arguments in the same pass:

```typescript
if (tc.id) {
  if (activeItemType) flushActiveItem();
  startToolCallItem(tc.index, tc.id, tc.function?.name || "");
}
// Always process arguments regardless of whether id was present
if (tc.function?.arguments) {
  const acc = toolCallAccum.get(tc.index);
  if (acc) {
    acc.args += tc.function.arguments;
    const item = outputItems[outputItems.length - 1];
    if (item?.type === "function_call") {
      item.arguments = acc.args;
    }
    enqueueSSE(controller, "response.function_call_arguments.delta", { ... });
  } else if (tc.id) {
    // Tool call was just started by the block above
    // First argument needs a fresh accumulator entry
    const acc = toolCallAccum.get(tc.index);
    if (acc) {
      acc.args = tc.function.arguments;
    }
  }
}
```

## Bug 4: HIGH — Empty content string in same chunk as `reasoning_content` triggers premature reasoning flush
**File:** `src/translate/stream/chat-completions-to-responses.ts`
**Lines:** 209-235 (reasoning handler), 265-287 (text handler)
**Severity:** HIGH

### Problem
Many reasoning models (DeepSeek, Qwen) send a priming chunk that contains BOTH `reasoning_content` AND `content: ""`:

```json
{"choices":[{"index":0,"delta":{"reasoning_content":"Let me think about this","content":""}}]}
```

Processing order:
1. Line 210: reasoning_content handler fires → `startReasoningItem()` → `activeItemType = "reasoning"` → reasoning delta emitted
2. Line 265: text content handler fires → `delta.content = ""` → `hasTextContent = true` → `activeItemType = "reasoning"` and `activeItemType !== "text"` → **`flushActiveItem()` is called** → reasoning item finalized prematurely
3. Line 271: empty content skipped

Then the next reasoning chunk:
```
{"choices":[{"index":0,"delta":{"reasoning_content":"more thinking"}}]}
```
- Creates a **second** reasoning item (incorrect)

**Result:** The output has TWO reasoning items instead of one. The first item has truncated reasoning text.

### Fix
The text handler should skip empty content if reasoning is currently active:

```typescript
const hasTextContent = delta.content !== undefined && delta.content !== null;
if (hasTextContent) {
  // Skip empty priming content when reasoning is active
  if (activeItemType === "reasoning" && delta.content === "") {
    // Don't flush reasoning for empty content
  } else if (activeItemType && activeItemType !== "text") {
    flushActiveItem();
  }
  // ... rest of text handling
}
```

## Bug 5: HIGH — Parallel tool call arguments in same chunk cross-contaminate
**File:** `src/translate/stream/openai-to-anthropic.ts`
**Lines:** 43-107 (tool call handler)
**Severity:** HIGH

### Problem
When two tool calls start in the same chunk (parallel tool calls):

```json
{"choices":[{"index":0,"delta":{"tool_calls":[
  {"index":0,"id":"call_abc","function":{"name":"foo","arguments":""}},
  {"index":1,"id":"call_def","function":{"name":"bar","arguments":""}}
]}}]}
```

1. Tool call "call_abc": `currentToolCallId = null` → enters `if` block → `currentToolCallId = "call_abc"`, `contentBlockIndex = 0`, content_block_start emitted
2. Tool call "call_def": `currentToolCallId = "call_abc"` → `"call_def" !== "call_abc"` → enters `if` block → **emits `content_block_stop`** for "call_abc" IMMEDIATELY (no deltas yet!), `currentToolCallId = "call_def"`, `contentBlockIndex = 1`, content_block_start emitted

Then for subsequent argument-only chunks (pattern common with parallel tool calls):
```json
{"choices":[{"index":0,"delta":{"tool_calls":[
  {"index":0,"function":{"arguments":"\"arg1\""}},
  {"index":1,"function":{"arguments":"\"arg2\""}}
]}}]}
```

3. Arguments for index 0: `toolCall.function?.arguments = "\"arg1\""` → truthy → `currentToolCallId = "call_def"` → arguments accumulated to **"call_def"** instead of "call_abc"! **Bug: cross-contamination**

### Root cause
`currentToolCallId` acts as a single "current" tool call pointer, but parallel tool calls require per-tool-call tracking. When two tool calls start in the same chunk, the code fails to properly separate them.

### Fix
Replace `currentToolCallId` with per-accumulator tracking keyed by tool call index or id, and differentiate between "new tool call" creation and "continuous argument accumulation" using more specific signals (e.g., presence of `id` field):

```typescript
// Track by tool call ID, and distinguish "new call creation" from "subsequent args"
if (toolCall.id) {
  // Always a new tool call — emit content_block_stop for PREVIOUS content type,
  // not for every tool call id switch
  if (!isToolUse && (hasStartedTextBlock || hasStartedThinkingBlock)) {
    enqueueSSE(controller, "content_block_stop", { ... });
    hasStartedTextBlock = false;
    hasStartedThinkingBlock = false;
    contentBlockIndex++;
  }
  if (!isToolUse) {
    // Emit content_block_start for the tool call
    // ... existing logic
  }
}
// Process arguments separately for all tool calls
if (toolCall.function?.arguments) {
  // Accumulate by toolCall.id, not by currentToolCallId
  const currentJson = toolCallJsonMap.get(toolCallId) || "";
  toolCallJsonMap.set(toolCallId, currentJson + toolCall.function.arguments);
  // Emit content_block_delta with appropriate index
}
```

## Bug 6: MEDIUM — Empty content string creates spurious empty text block before reasoning
**File:** `src/translate/stream/openai-to-anthropic.ts`
**Lines:** 157-202 (text content handler)
**Severity:** MEDIUM

### Problem
When a priming chunk contains both `reasoning_content: ""` AND `content: ""`:

```json
{"choices":[{"index":0,"delta":{"reasoning_content":"","content":""}}]}
```

1. `delta.reasoning_content = ""` → falsy → thinking handler skipped
2. `delta.content = ""` → `delta.content !== undefined && delta.content !== null` → **true** → text handler fires
3. Empty text block is created: `content_block_start(type: "text", text: "")` + `content_block_delta(text: "")`
4. Next chunk with `reasoning_content: "thinking"` causes text→thinking transition, emitting `content_block_stop` for the spurious text block

**Result:** The stream has an extra empty text block at position 0 before the thinking block:
```
content_block_start(index=0, type="text", text="")   ← spurious
content_block_delta(index=0, type="text_delta", text="")  ← spurious
content_block_stop(index=0)                            ← spurious
content_block_start(index=1, type="thinking", ...)     ← correct
content_block_delta(...)                               ← correct
```

While not technically invalid, this wastes bandwidth and may confuse clients that validate block order.

### Fix
Add a check for empty content at the text handler entry:

```typescript
if (delta.content !== undefined && delta.content !== null) {
  if (delta.content === "" && !hasStartedTextBlock) {
    // Skip empty priming content — don't create a text block for nothing
    return;
  }
  // ... rest of handler
}
```

Or better, defer text block creation until non-empty content is received.

## Bug 7: MEDIUM — Tool call indices use Anthropic content block index instead of 0-based tool call sequence
**File:** `src/translate/stream/anthropic-to-openai.ts`
**Lines:** 74-81 (tool call emission), 97-101 (tool call argument emission)
**Severity:** MEDIUM

### Problem
Tool call indices are set to the Anthropic content block index (`contentBlockIndex = evt.index`), which may be offset if text blocks precede tool calls.

**Anthropic stream example:**
```
content_block_start(index=0, type="text")
content_block_delta(text="Hello")
content_block_stop(index=0)
content_block_start(index=1, type="tool_use", id="call_abc")
content_block_delta(input_json_delta, partial_json='{...}')
content_block_stop(index=1)
```

**Current output:**
```
data: {"tool_calls":[{"index":1,"id":"call_abc","function":{"name":"...","arguments":""}}]}
```

**Expected output (OpenAI convention):**
```
data: {"tool_calls":[{"index":0,"id":"call_abc","function":{"name":"...","arguments":""}}]}
```

The OpenAI SDK uses the `index` field to correlate argument deltas across chunks. While a non-zero starting index may work if consistent, OpenAI's own API always starts at 0. Some client implementations may fail with non-standard index values.

### Fix
Maintain a separate `toolCallIndex` counter independent of `contentBlockIndex`:

```typescript
let toolCallIndex = 0;

// When tool_use content_block_start fires:
emitChunk({
  tool_calls: [{
    index: toolCallIndex++,
    id: tcId,
    type: "function",
    function: { name: block.name || "", arguments: "" },
  }],
});

// When input_json_delta fires:
emitChunk({
  tool_calls: [{
    index: toolCallIndex - 1,  // index of the current tool call
    function: { arguments: delta.partial_json || "" },
  }],
});
```

## Bug 8: MEDIUM — Stream ending with only empty content and stop reason produces empty output array
**File:** `src/translate/stream/chat-completions-to-responses.ts`
**Lines:** 271-273 (empty content skip), 354-380 (final response)
**Severity:** MEDIUM

### Problem
When the only non-null delta content is an empty string with a stop reason:

```
data: {"choices":[{"index":0,"delta":{"content":""},"finish_reason":"stop"}]}
```

Processing:
1. `delta.content = ""` → `hasTextContent = true`
2. `activeItemType = null` → first-time empty priming logic → skipped (line 271)
3. `delta.content = ""` → falsy → no delta emission

At stream end:
- `activeItemType = null` → no flush
- `finishReason = "stop"` → `status = "completed"`
- `outputItems = []` (empty!)

**Result:** Final response has `output: []`, which means the response appears to have no content. The original Chat Completions chunk had one choice with empty content, so there should be at least one output item.

### Fix
Handle the degenerate case where content is empty but a finish_reason exists:

```typescript
if (delta.content === "" && !activeItemType) {
  // Empty priming chunk — but create text item if this is the final chunk
  if (parsed.choices?.[0]?.finish_reason) {
    // This is an empty final response — create an empty text item
    startTextItem();
    textAccum = "";
    // Don't emit text delta since content is empty
  } else {
    // Don't create text item yet
  }
}
```

Or more robustly, at stream end, if no output items were created and finishReason is present, create a synthetic empty text item.

## Bug 9: LOW — [DONE] marker sent unconditionally even after stream errors
**File:** `src/translate/stream/anthropic-to-openai.ts`
**Lines:** 140-168 (try-finally block), 185 ([DONE] emission)
**Severity:** LOW

### Problem
The code wraps the reading loop in `try { ... } finally { reader.releaseLock(); }`, but does NOT catch errors. If the upstream stream throws (network error, premature close), the `finally` block runs, the reader is released, and execution continues to line 185 where `[DONE]` is emitted and `controller.close()` is called.

This means upstream errors are silently masked — the client receives:
```
data: [DONE]
```

as if the stream completed normally, even though the payload was truncated or an error occurred.

### Fix
Wrap the read loop in `try/catch`. On error, either close the controller with an appropriate error:

```typescript
try {
  while (true) { /* ... */ }
} catch (err) {
  // On error, close without [DONE] to signal abnormal termination
  controller.close();
  reader.releaseLock();
  return;
} finally {
  reader.releaseLock();
}
```
